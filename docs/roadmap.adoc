= Lapis Net Roadmap

== V0.1 — Protocol core + Veritas

Target: first runnable prototype — network bootstrap plus Veritas (the first
of the four scoring dimensions), including GossipSub propagation of trust
edges between local nodes.

[cols="1,3"]
|===
| Wave | Content

| V0.1.1
| Gradle multi-module skeleton + CI/CD skeleton.

| V0.1.2
| Identity layer: secp256k1 keypair generation/persistence, dual-key binding to Ed25519 for the libp2p peer ID.

| V0.1.3
| P2P node bootstrap: jvm-libp2p setup, bootstrap peers, local mDNS discovery.

| V0.1.4
| Nabu integration: DHT + Bitswap for content storage (Veritas records as signed files).

| V0.1.5
| Veritas core: trust-edge data model, signing, Git-like versioning (backward hash to the previous grant).

| V0.1.6
| Veritas path algorithm: shortest paths, lexicographic ordering by next hop, product of edge weights.

| V0.1.7
| GossipSub propagation: distribute Veritas records over a dedicated topic and persist them in Nabu.

| V0.1.8
| Minimal CLI / test harness: demo with N local nodes, trust-edge issuance and propagation made visible.
|===

=== Progress

V0.1.1 through V0.1.8 are implemented (see `docs/architecture.adoc` for
details). **The V0.1 milestone is complete.**

== V0.2 — Virtus + Minimal-Browser

Target: the second of four scoring dimensions — Virtus (the LTR/on-chain
economy) — plus a Minimal-Browser that makes Veritas and Virtus data
actually visible. The browser is deliberately placed here rather than as a
standalone later wave, so a real pilot test (PdV, Girchi) is possible as
soon as V0.2 lands, instead of only after every scoring dimension exists.

[cols="1,3"]
|===
| Wave | Content

| V0.2.1
| Virtus/LTR core (on-chain): `LtrRecord` data model, GossipSub propagation
+ Nabu persistence (mirroring the Veritas V0.1.5/V0.1.7 pattern), 10%/24h
decay math, accumulation (independent records per (cid, view) pair, never
merged). Structural-only `OnChainProof` — no live Bitcoin chain
verification yet (known, documented limitation).

| V0.2.2
| Minimal-Browser (MVP): a local, loopback-only Ktor HTTP server embedding
a real node, serving a framework-free HTML/JS timeline UI. Shows Veritas
credibility + Virtus/LTR-weighted sorting; a small, deliberately temporary
`PostAnnouncement` primitive fills the content-discovery gap Karma (V0.3)
will eventually close properly.
|===

=== Progress

V0.2.1 and V0.2.2 are implemented (see `docs/architecture.adoc`'s "Virtus:
LTR core (V0.2.1)" and "Minimal-Browser MVP (V0.2.2)" sections). **The V0.2
milestone is complete.**

== V0.3 — Karma

Target: the third of four scoring dimensions — Karma, a cheap one-click
"like" with Bitcoin-chain-anchored, Veritas-weighted Sybil resistance —
wired into the Minimal-Browser as a persistently-visible Like button.

[cols="1,3"]
|===
| Wave | Content

| V0.3
| `lapis-net-karma`: `KarmaVote` data model (`Karma = t/(n+1)`, `t` =
blocks since the voter's own first outgoing Bitcoin transaction, `n` =
the voter's own prior eligible-vote count, always locally derived, never
self-declared); GossipSub propagation with zero live chain-verification
in the validation path (mirroring `OnChainProof`'s V0.2.1 precedent); a
client-side, trusting (not trustless-SPV) Electrum chain-data source used
only at local vote-cast time. Wired into `lapis-net-browser`: a new
`POST /api/karma` route and a Like button, with Karma kept purely
additive/display-only in `TimelineBuilder` (never affecting sort or
filtering).
|===

=== Progress

V0.3 is implemented (see `docs/architecture.adoc`'s "Karma: Bitcoin-
anchored one-click likes (V0.3)" section). **The V0.3 milestone is
complete.** Three of the fach-spec's four scoring dimensions (Veritas,
Virtus, Karma) are now implemented; Madli remains the only one not yet
started.

== V0.4 — Genesis-Bootstrap + Identity Hardening

Target: move from test fixtures toward real users. Bundles four related
items that all circle the same theme: real bootstrap configuration, a
QR-code/deep-link key exchange flow, multi-device/pseudonym self-trust
linking, and keystore encryption at rest.

[cols="1,3"]
|===
| Wave | Content

| V0.4
| `BootstrapConfig` (`lapis-net-networking`): env-var + file-based
bootstrap-peer resolution and a configurable libp2p listen address,
replacing the old RFC-5737 placeholder list. `ConnectUri`/`QrCodeSvg`
(`lapis-net-browser`): a `lapisnet://connect` URI scheme plus a
server-side-rendered SVG QR code and three new HTTP routes so two real
users can connect without hand-copying multiaddrs. `POST
/api/self-link`: multi-device/pseudonym linking that reuses the
existing `VeritasGrant` model verbatim at maximum trust — no schema
change, no new Sybil attack surface. `KeystoreEncryption`/`KeystoreFileFormat`
v2 (`lapis-net-identity`): Argon2id + AES-256-GCM keystore encryption
at rest with the full file header bound as GCM AAD, and transparent,
atomic v1-to-v2 migration.
|===

=== Progress

V0.4 is implemented (see `docs/architecture.adoc`'s "Genesis-Bootstrap +
Identity Hardening (V0.4)" section). **The V0.4 milestone is complete.**
