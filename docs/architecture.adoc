== Layering

Lapis Net separates a mandatory protocol core from an optional scoring
substrate and a fully pluggable application layer:

* *Protocol core* (mandatory for every participant): identity (secp256k1),
  content addressing, storage (Nabu/Bitswap), peer discovery (DHT),
  propagation (GossipSub), Lightning payments.
* *Scoring substrate* (optional): Veritas, Karma, Madli as network-wide
  reputation records in the DHT; Virtus/LTR as view-specific records bound to
  a browser vendor.
* *Application layer* (freely swappable per vendor): browser/view
  implementations, sorting, scoring interpretation, UI, ad policy.

The network functions even with zero scoring systems in use — Veritas,
Virtus, Karma, and Madli are tools for views, not a protocol requirement.

== Identity (V0.1.2)

The canonical Lapis Net identity is a **secp256k1 keypair** (`secp256k1-kmp`) — the same key type
that would back a Bitcoin wallet or Lightning channel in a later wave. There is no recovery
mechanism: losing the private key means losing the identity permanently, by design, analogous to
Bitcoin.

libp2p natively wants an **Ed25519** keypair for its peer ID. Since secp256k1 is the
real identity, each identity carries a second, Ed25519 keypair (Bouncy Castle,
`bcprov-jdk18on` — the same library jvm-libp2p itself depends on) used only for the libp2p peer
ID, plus a **dual-key binding**: the secp256k1 key signs a domain-separated digest of the Ed25519
public key, cryptographically proving "this libp2p peer ID belongs to this Lapis Net identity"
without making Ed25519 the canonical identity.

Domain separation (`net.lapisphilosophorum.lapisnet.core.crypto.domainSeparatedDigest`) is a
project-wide convention, not specific to identity binding: every distinct purpose that signs a
32-byte digest with a secp256k1 key (identity binding today; Veritas trust-edge grants from
V0.1.5) must use its own versioned ASCII tag, so a signature produced for one purpose can never be
confused with a signature for another purpose using the same key.

Identities persist to a fixed-layout binary keystore file (`KeystoreFileFormat`, one file per
label under `$LAPISNET_HOME/identity` or `~/.lapisnet/identity`), protected only by POSIX file
permissions (`0600` files, `0700` directory) — the same trust model `ssh-keygen` uses by default.
This is a deliberate, temporary tradeoff for an early prototype: passphrase-derived encryption at
rest is a documented follow-up that should land before this key custodies real funds (Lightning).
A user may hold several identities (pseudonyms) side by side, addressed by label — device-sync /
self-trust linking between a user's own pseudonyms is a later wave (see the project's Veritas
notes), not implemented here.

== Networking (V0.1.3)

The node's libp2p identity is derived directly from `DualKeyIdentity`'s Ed25519 keypair
(`deriveLibp2pPeerId()`): the raw 32-byte seed V0.1.2 already stores is exactly the byte layout
jvm-libp2p's key unmarshalling expects, so no adapter or re-encoding is needed. The `PeerId` is
derived from the *private* key (via its public key), never read from a separately-stored public
key, so a corrupted keystore can never desync the `PeerId` from what the node can actually sign
with.

`LapisNode` wraps a jvm-libp2p `Host` built from the production-tested combination the library's
own status table recommends: TCP transport, Noise secure channel, mplex stream muxer. mDNS
discovers LAN peers but never auto-dials them — jvm-libp2p's own `Discoverer` API is
discovery-only by design, and keeping discovery and connectivity independently observable makes
both easier to test. Bootstrap peers are the opposite case: dialing them on `start()` is the
whole point, done non-blockingly and non-fatally so an unreachable peer can never hang or fail
startup. No real bootstrap infrastructure exists yet — `BootstrapPeers.PLACEHOLDER` uses RFC 5737
documentation-range addresses, explicitly marked as a pre-deployment placeholder.

jvm-libp2p is not published to Maven Central; it ships across three mirrors (Cloudsmith, JitPack,
Consensys artifacts) that must all be declared as repositories, since its own transitive
dependencies (`java-multibase`, `noise-java`) are split across them.

GossipSub pub/sub and NAT traversal are explicitly out of scope for this wave; GossipSub lands in
V0.1.7 (see below), NAT traversal remains an open research item.

== Storage (V0.1.4)

`NabuStorage` (in `lapis-net-storage`) layers Nabu's DHT (Kademlia) and Bitswap protocols onto
the same `Host` a `LapisNode` already built and started in V0.1.3 (`LapisNode.host` is exposed,
read-only, specifically for this). Its API is domain-agnostic by design — `ByteArray`/`Cid` in
and out — so it can serve as the storage substrate for signed Veritas records (V0.1.5+) without
knowing anything about Veritas itself.

**Classpath collision with Nabu's own forked jvm-libp2p.** Nabu's `pom.xml` pins a transitive
dependency on `com.github.peergos:jvm-libp2p:0.18.0-ipv6-mdns-wildcard`, a Peergos-maintained fork
of the official `io.libp2p:jvm-libp2p` this project already depends on. The fork ships identical
Java package names (`io.libp2p.core.*`) but different bytecode for 361 of 829 shared classes
(verified by decompiling both jars). `lapis-net-storage/build.gradle.kts` excludes the fork and
forces Nabu to resolve against the official artifact instead — verified safe because Nabu's
`Bitswap`/`Kademlia` classes (and their engines) have zero compiled references to the
`io.libp2p.core.dsl.*` builder classes that actually differ between the two jars (only Nabu's own
unused `org.peergos.HostBuilder` touches those), confirmed both statically and by a real two-node
Bitswap put/get spike against the official jar. `NabuStorage` therefore always attaches to a
`Host` built via the official DSL (`LapisNode.create()`), never via Nabu's own `HostBuilder` or
`EmbeddedIpfs`.

**Access control: none yet, by design.** `NabuStorage.attach()` wires an always-allow
`BlockRequestAuthoriser` — any connected peer can fetch any block this node has stored, provided
or not, the standard Bitswap/IPFS trust model (CIDs are unguessable 256-bit content hashes; a peer
cannot induce this node to fetch-and-forward third-party content, since `get()`/`findProviders()`
are only called by this project's own local code, never in response to a remote request).
Per-caller/per-record access control is a Veritas-layer concern and lands in V0.1.5+, not here.

**DHT record-store bound.** Nabu's own `RamRecordStore` (used for IPNS-style key/value records)
is an uncapped `HashMap`, and its `KademliaEngine` hands any connected peer's signed `PUT_VALUE`
record straight into it with no other rate limit - reachable simply by wiring up Kademlia at all,
regardless of whether this project's own API ever publishes an IPNS record itself.
`lapis-net-storage`'s own `BoundedRecordStore` (LRU-capped, mirroring Nabu's own
`RamProviderStore`) is used instead to prevent unbounded remote-triggered heap growth.

**Known limitation: cross-node DHT provider discovery does not currently work.**
`NabuStorage.provide()`/`findProviders()` wrap Nabu's real `Kademlia.provideBlock`/
`findProviders` API correctly (verified via `javap` against the real jar), and the underlying
DHT routing-table bootstrap (`connectToDhtPeer`, wrapping `Kademlia.bootstrapRoutingTable`) is
verified working — a 3-node loopback chain reliably populates each node's routing table with its
explicitly-connected peer. However, the actual provider announcement/discovery round trip does
not: a standalone diagnostic spike (four isolated runs, outside the test suite, against real
loopback nodes) found that a `GET_PROVIDERS` RPC returns an empty provider list even via Nabu's
own trivial "I already have this block locally" auto-provide path in
`KademliaEngine.receiveRequest`'s `GET_PROVIDERS` case — i.e. this reproduces with zero
involvement of `provide()`/`ADD_PROVIDER` at all, so the fault is in the RPC/dial mechanism
`Kademlia` uses for both message types (`Kademlia.dialPeer`), not specific to provider
announcement. The failure is reproducible, not flaky (unlike V0.1.3's mDNS multicast issue,
which has a genuine external cause) — root cause not identified within this wave's practical
debugging budget; suspected candidates include an interaction between `/ipfs/` vs. `/p2p/`
peer-id multiaddr component naming (routing-table entries carry `/ipfs/`-form addresses) and
`dial()`'s peer-id validation, or a Nabu v0.8.0-specific issue. `lapis-net-storage`'s test suite
pins this with an explicit "KNOWN ISSUE" test
(`MultiNodeDhtProviderDiscoveryTest`) asserting the current (broken) empty result, so a future
fix shows up as a test failure prompting an update, rather than the regression going unnoticed.
What remains fully verified and usable for V0.1.5+: local put/get, and Bitswap fetch given an
explicit peer (the `TwoNodeBitswapDirectFetchTest` path) — i.e. `get(cid, peers = ...)` with a
peer set obtained by some other means (a future wave's own peer-list, GossipSub-announced peers,
etc.) is safe to build on; relying on `findProviders()` to return non-empty is not, until this is
root-caused.

== Trust: Veritas trust-edge grant (V0.1.5)

`VeritasGrant` (in `lapis-net-trust`) is the signed, versioned data model for a single Veritas
web-of-trust edge: a truster's secp256k1-signed statement of how much it trusts a target identity.
It is a pure data-model wave — network propagation (GossipSub) and persistence (Nabu) are
implemented in V0.1.7, below; the shortest-path aggregation algorithm that turns a graph of grants
into a trust score between two arbitrary identities is implemented in V0.1.6, below.

**Fixed-point trust value.** Trust is represented as `trustMicros: Int` in `0..1_000_000`, not as a
`Double`. A signed, cross-platform-verified structure cannot depend on floating-point arithmetic
being bit-identical across JVMs/architectures; a fixed-point integer sidesteps that class of bug
entirely. `trustFraction` (`trustMicros / 1_000_000.0`) exists only as a display-only convenience,
never as part of the signed or compared representation.

**Single target per grant.** Each `VeritasGrant` names exactly one `(truster, target)` pair, not a
batch of targets, because the Git-like backward-hash version chain (below) is intrinsically
per-pair: a grant's `previousGrantId` must unambiguously identify the one prior grant it supersedes
for that same truster/target combination.

**Git-like backward-hash version chain.** Each new grant optionally embeds the content hash
(`VeritasGrantCodec.contentId()`, plain SHA-256 over the full signed-and-signature-appended
encoding) of the grant it supersedes, as `previousGrantId`. The first grant in a chain (no
predecessor) is genesis, represented as a `null` `previousGrantId` with an explicit presence flag
bit in the wire encoding — not a sentinel all-zero hash. Revocation is not a distinct operation or
message type; it is simply a new grant with `trustMicros = 0` that links to its predecessor like
any other update. `VeritasGrantChain.validate()` checks an ordered list of grants forms a valid
chain (constant truster/target, first is genesis, every signature verifies, each grant's
`previousGrantId` matches its predecessor's `contentId()`) — purely in-memory, no I/O; fetching the
actual chain from storage is a V0.1.7 concern.

**Domain-separated signing.** The signed body is digested with
`domainSeparatedDigest("LapisNet:veritas-trust-edge:v1", body)` before signing — its own reserved,
versioned tag distinct from identity binding's `"LapisNet:identity-binding:v1"`, per this project's
domain-separation convention (see Identity above): a signature produced for one purpose can never
be confused with a signature for another purpose using the same key.

**Codec.** `VeritasGrantCodec` uses a sequential length-prefixed binary layout (magic, version,
flags, truster/target public keys, `trustMicros`, optional `previousGrantId`, length-prefixed
comment, length-prefixed occasion-reference CIDs, signature) rather than `KeystoreFileFormat`'s
fixed-offset layout, since several fields are variable-length. `decode()` is structural-only and
deliberately does **not** verify the signature — unlike `KeystoreFileFormat.decode()`, which must,
because a keystore has to be trustworthy just to load. A grant typically arrives from an untrusted
peer over the network, so callers must explicitly call `VeritasGrant.verify()` (or the overload
taking an expected truster) before trusting a decoded grant.

**Security fix: secp256k1 public-key curve validation.** Independent security review for this wave
found that `Secp256k1PublicKey` (in `lapis-net-identity`, present since V0.1.2) validated only that
its input was 33 bytes long, never that those bytes actually encode a point on the secp256k1
curve. The native `Secp256k1.verify()` call throws an uncaught exception on a curve-invalid public
key rather than returning `false`, and does so before any signature check — no valid signature
knowledge is needed to trigger it. This had gone unnoticed through V0.1.2–V0.1.4 because nothing
until this wave constructed a `Secp256k1PublicKey` directly from untrusted, decoded bytes;
`VeritasGrantCodec.decode()` is the first code path that does, making this a real remote-DoS vector
once grants start arriving from peers (V0.1.7). Fixed by validating via `Secp256k1.pubkeyParse()`
in the constructor, converting any native failure into a clean `IllegalArgumentException`; the fix
applies project-wide, since the class is shared. `VeritasGrantCodec.decode()` was also hardened to
reject non-zero reserved flag bits, so tampered bits can no longer silently round-trip as an equal
grant.

== Trust: transitive path selection (V0.1.6)

`TrustGraph` and `TrustPathFinder` (in `lapis-net-trust`) compute a transitive trust score and
winning path between a source and a target identity, over a graph of `VeritasGrant`-derived edges.
This wave is pure algorithm: it consumes an already-resolved edge set (exactly one latest grant per
`(truster, target)` pair) and does not itself fetch grants from storage or resolve a grant's
backward-hash version chain to its latest entry — both are implemented in V0.1.7, below.

**Selection rule: shortest path first, then lexicographic tiebreak — not maximum product.** Trust
along a path is the product of its edge weights, but when multiple shortest (fewest-hop) paths
exist, the winner is not the one with the highest product; it is the one whose hop-weight sequence,
compared from the source outward, is lexicographically greatest (highest first hop wins; ties are
broken by the second hop, then the third, and so on). This models "the closer a person is to me,
the better I can judge them" — a strong nearby relationship outweighs a clever construction of
distant ones, and it is also manipulation-resistant: an attacker cannot inflate transitive trust by
stacking high weights far out in the graph, since only the near edges are ever decisive. The
project's canonical worked example (Odin → Frigg/Thor/Nanna → Freya, where the highest-product path
via Nanna does *not* win because its first hop is weaker) is reproduced verbatim as a binding test
fixture. Self-trust (`source == target`) is axiomatic (`1.0`); no path is `null` (default distrust,
`0`); a zero-trust edge is a real hop that can legitimately be part of, and even win, the shortest
path — there is no fallback to a longer, higher-trust alternative, matching the spec's explicit
single-winner, no-aggregation design for V0.1.

**Fixed-point score, `Long` accumulator.** The path score is computed as `trustMicros: Int` in the
same fixed-point domain as `VeritasGrant.trustMicros`, via a `Long` intermediate accumulator
(`acc = acc * weight / MAX_TRUST_MICROS` at each hop) — two `Int` values up to `1_000_000` multiply
to roughly `10^12`, well past `Int.MAX_VALUE`, so the accumulator must widen even though the final
result always fits back into an `Int`.

**BFS with predecessor pointers, not eager path copies.** Finding "shortest path" means fewest
hops, not lowest weighted distance — this is plain level-synchronous BFS, not Dijkstra. The
algorithm's first implementation stored a full copy of the path-so-far (node list + weight list) at
every node reached during the traversal; on a simple linear chain of `N` nodes this cost `O(N^2)`
time and memory for a single query, and since Veritas grants are free to issue and self-signable in
bulk, an attacker needed no cooperation from anyone else to construct such a chain — a real
remote-DoS vector for the moment gossiped/stored grant data starts feeding real queries (V0.1.7).
The fix replaces the eager copy with a predecessor-pointer scheme: each node stores only a pointer
to its winning predecessor plus the single edge weight into it (`O(1)` per node), and the winning
path is reconstructed exactly once, by walking predecessor pointers backward from the target, only
after BFS finds it. The lexicographic tiebreak (needed when multiple predecessors reach the same
node at the same BFS depth) is implemented as a pair of small recursive comparators that walk
already-finalized predecessor chains, comparing the earliest (most significant) hop first with
early exit on the first difference, rather than comparing pre-built arrays.

**Constant-factor hardening.** A follow-up independent security audit found that while the
predecessor-pointer redesign is asymptotically correct (`O(edges * MAX_HOP_DEPTH)`, confirmed by an
independent hand-trace), the constant factor at the original defense-in-depth caps
(`MAX_NODES = 50_000`, `MAX_EDGES = 200_000`) was not negligible: an adversarial graph where every
edge shares the same weight forces every lexicographic comparison to recurse through the full
predecessor chain before resolving (via a node-public-key-bytes tiebreak) instead of short-
circuiting on the first differing hop — costing several hundred milliseconds per query at those
caps, cheaply reachable since the attacker only needs free, self-signed grants. Two changes
resolved this: `Secp256k1PublicKey.hashCode()` (in `lapis-net-identity`) is now cached at
construction rather than recomputing a 33-byte content hash on every call — safe because the class
is immutable, and a meaningful win since every recursive predecessor-chain comparison hits it via
`HashMap` lookups; and the caps were tightened to `TrustGraph.MAX_NODES = 8_000`,
`TrustGraph.MAX_EDGES = 32_000` (`TrustPathFinder.MAX_HOP_DEPTH` stayed at `20`, itself independently
justified by trust decay: even a generous uniform `0.9` weight per hop decays to `0.9^20 ≈ 0.12` by
that depth, so longer paths are already close to meaningless as trust signals). Re-measured against
the worst adversarial construction found across two independent audit rounds — a tie-heavy graph
shifted to the maximum recursion depth the hop cap allows, not just concentrated at the first two
hops — worst-case cost at the new caps is roughly 35-50ms, an order of magnitude better than at the
original caps and comfortably within the caps' own generous-for-personal-scale headroom (Dunbar's
number is roughly 150; 8,000 nodes is far beyond any realistic personal web-of-trust).

This wave went through three rounds of independent review/security audit before landing — the
correctness of the BFS/tiebreak algorithm itself was hand-traced and approved in the first round and
never needed to change again; both follow-up rounds addressed resource-exhaustion cost at scale, not
correctness.

== Trust: propagation and persistence (V0.1.7)

`GossipPubSub` (in `lapis-net-networking`) and `VeritasGossip`/`VeritasGrantIndex`/
`VeritasGrantResolver` (in `lapis-net-trust`) close the loop V0.1.3-V0.1.6 left open: a node can now
announce a locally-created `VeritasGrant` to its peers over a dedicated GossipSub topic, receive and
validate grants announced by others, persist them into `NabuStorage`, and resolve the "latest grant
per `(truster, target)` pair" edge set that `TrustGraph.fromGrants()` expects.

**Layering.** `GossipPubSub` is domain-agnostic (raw topic strings, `ByteArray` in/out) and lives in
`lapis-net-networking`, layered onto an already-`start()`-ed `LapisNode`'s `Host` exactly the way
`NabuStorage` layers Bitswap/Kademlia onto the same `Host` in V0.1.4. `VeritasGossip` is the
Veritas-specific wiring on top: topic naming, structural decode, signature verification, and the
persistence/resolution logic. `lapis-net-trust` gained a new `api(project(":lapis-net-storage"))`
dependency this wave, since `NabuStorage` now appears directly in `VeritasGossip`'s public factory
signature.

**Two separate signature layers that must never be confused.** GossipSub has its own message-layer
authentication (the `from`/`seqno`/`signature` RPC fields), signed with the node's libp2p Ed25519
identity key (`Host.privKey`) — this is entirely separate from a `VeritasGrant`'s secp256k1 truster
signature, and `GossipPubSub` never touches the latter.

**Full grant bytes are published, not a CID pointer.** A `VeritasGrant`'s encoded size
(`VeritasGrantCodec.MAX_BODY_SIZE` plus a 64-byte signature) is tiny relative to any reasonable
gossip message-size ceiling, and — decisively — `NabuStorage`'s documented V0.1.4 known limitation
is that cross-node DHT provider discovery (`findProviders()`) does not work: a node receiving only a
bare CID over gossip would have no reliable way to fetch the actual content. Publishing full bytes
sidesteps that broken discovery path entirely.

**The GossipSub validator gates both structural and cryptographic acceptance in one step.**
`VeritasGossip.onGossipMessage()` decodes (`VeritasGrantCodec.decode()`), then verifies
(`VeritasGrant.verify()`), before any side effect — deliberately not split into "validate then
separately accept", since GossipSub's own re-propagation to the rest of the mesh is gated purely by
the validator's returned `ValidationResult`; deferring signature verification to a later step would
mean every relaying node re-gossips signature-invalid garbage before individually declining to act
on it.

**Latest-grant-per-pair resolution.** `VeritasGrantResolver.resolveLatest()` collapses a (possibly
forked, possibly byzantine) set of candidate grants for one `(truster, target)` pair down to a
single winner: only signature-valid candidates are considered; every valid genesis grant is a root;
an iterative depth-first walk (explicit stack, not recursion, to avoid stack-overflow risk at
`MAX_CHAIN_DEPTH = 4_096`) follows the backward-hash chain forward from each root, tracking a
visited set to guard against (cryptographically infeasible, but defended anyway) cycles; the winner
among all resulting tip candidates is the one at the greatest depth, tied-broken by
lexicographically-greatest content-id bytes. This tiebreak is deliberately receipt-order-independent
— it depends only on the final candidate set, never on the order candidates arrived via gossip — so
two nodes that eventually observe the same grants via differently-ordered deliveries converge on the
identical resolved graph.

**Known, deliberately accepted gaps** (matching this project's established practice of documenting
limitations rather than silently shipping them):

* The in-memory `VeritasGrantIndex` is never rebuilt by scanning Nabu's local blockstore at startup
  — `NabuStorage` has no "enumerate local CIDs" primitive. A restarted node starts with an empty
  index regardless of what it durably persisted in a previous run.
* A node joining after a grant's gossip window has passed has no backfill path — gossip carries full
  bytes but only propagates live, never historically. This is best-effort convergence, not
  always-eventually-consistent.
* Unbounded message *rate* from a single peer is not defended against at the application layer —
  deferred to jvm-libp2p's own GossipSub mesh-control-message rate limiting.

**This wave went through four rounds of independent review/security audit — the most of any wave so
far** (V0.1.6's three-round precedent was the previous high-water mark), because it's the first wave
where `VeritasGrant`s actually arrive from untrusted network peers rather than being constructed
locally for tests:

* *Critical:* `VeritasGossip.onGossipMessage()` called `VeritasGrant.verify()` unguarded. A
  structurally-right-length but cryptographically degenerate 64-byte signature (e.g. all-`0xFF`
  bytes, or an `r` component at/above the secp256k1 curve order) makes the native
  `Secp256k1.verify()` call throw an uncaught exception instead of returning `false` — the same bug
  class as V0.1.5's `Secp256k1PublicKey` curve-validation fix, but that fix only ever hardened
  public-key *construction*, never signature *verification*. Live-repro'd against the real
  `secp256k1-kmp-jvm` jar. Fixed at the shared root cause — `Secp256k1PublicKey.verify()` (in
  `lapis-net-identity`) now wraps the native call in a catch-all, converting any failure into a
  clean `false` — rather than patching just the new call site, so every current and future caller is
  safe by construction.
* *Critical:* `NabuStorage.put()` ran unconditionally for every signature-valid grant, and
  `VeritasGrantIndex.add()`'s return value was discarded — meaning the index's capacity cap bounded
  neither disk writes nor GossipSub re-propagation (`ValidationResult.Valid` is what triggers every
  relaying node's own mesh-wide re-broadcast). Since Veritas grants cost nothing to issue, a cheap
  attacker could force unbounded, network-amplified disk growth across every node in the mesh.
* *Major:* `GossipPubSub.attach()` left GossipSub's own RPC-frame-shape limits
  (`maxPublishedMessages` etc.) at jvm-libp2p's actual default of "no limit" (confirmed by
  decompiling the real jar) — one peer could pack thousands of distinct grants into a single RPC
  frame, forcing unbounded decode+verify work per frame. Fixed by explicitly bounding
  `maxPublishedMessages`/`maxSubscriptions` via `GossipParamsBuilder`.
* *Major:* `VeritasGrantIndex` had no eviction once full — a cheap attacker filling the shared cap
  could permanently prevent a node from ever tracking new legitimate grants, including a legitimate
  revocation of a compromised or malicious truster (per the domain model, revocation is just a new
  grant with `trustMicros = 0`, not a distinct operation). Fixed with LRU-style eviction, mirroring
  `lapis-net-storage`'s `BoundedRecordStore` pattern.
* *The most interesting finding:* fixing the eviction gap (above) silently reopened the disk-growth
  gap (above it) — once the index stopped hard-rejecting at capacity and started evicting instead,
  its `canAccept()` check no longer had any "index is full" case left to report, so gating
  `storage.put()` on it alone stopped bounding persistence at all. This was caught independently by
  two review agents via two different methods (a hand-trace and an executed repro with `maxTracked =
  2` and 10 distinct grants, all 10 of which landed on disk) converging on the identical finding in
  the same round. Resolved by decoupling the two concerns: the in-memory trust-view index
  (`grantsByContentId`/`grantsByPair`, read exclusively by `TrustGraph`/`currentGrants()`) keeps its
  evicting policy — never permanently locking out legitimate updates is the correct policy for a live
  view — while a new, separate, non-evicting, hard-capped structure (`persistedContentIds`, gated via
  `VeritasGrantIndex.tryReservePersistence()`, cap `MAX_PERSISTED_GRANTS = 64_000`) independently
  bounds whether a durable disk copy gets written. A grant that clears the persistence cap still gets
  indexed and still propagates — this node's own disk budget has no bearing on trust-view correctness
  or mesh-wide convergence, since neither ever reads from disk.

== CLI / multi-node demo (V0.1.8)

`lapis-net-cli`'s `main()` runs two demos in sequence: `runIdentityAndStorageDemo()` (V0.1.4-era
single-node identity + local storage round-trip, unchanged in substance, only extracted into its own
function this wave) and `runMultiNodeTrustPropagationDemo()` — new this wave, the first piece of code
in the repository that exercises the full V0.1.2-V0.1.7 stack together in one running scenario rather
than each module's own isolated test suite.

**Topology.** Three in-process nodes are chained `A<->B<->C`; `A` and `C` are never directly dialed,
deliberately mirroring `ThreeNodeVeritasGossipRelayTest`'s topology (chosen there, and reused here,
specifically because it proves GossipSub *relay* through an intermediate node, not just direct
delivery). Node `A` issues a Veritas trust-edge grant to node `C`'s own secp256k1 identity; the grant
is gossiped through relay node `B` and resolved into a trust score on node `C`.

**Network hops vs. trust-graph hops.** The demo explicitly narrates a distinction that is easy to
conflate: the grant travels the network as 2 hops (`A->B->C`), but `TrustPathFinder`'s resolved trust
path is 1 hop (`A->C` directly) — the grant is a single signed `A->C` statement, and `B` only relays
the bytes; it never becomes a node in the trust graph itself. The test suite (below) pins this
distinction with an exact assertion (`hops shouldBe 1`), not just "didn't throw".

**Warm-up phase.** Before the real grant-announce round, the demo runs a disposable warm-up topic
where every message always validates `Valid`, so retried payloads keep getting relayed by `B`
regardless of duplicate content — unlike the real Veritas topic's content-addressed dedup. This is
not optional: skipping it makes the 3-node case flaky in a way a 2-node case never is, because grant
dedup means a lost mesh-formation race is not recoverable by simply retrying identical grant bytes.
See `ThreeNodeVeritasGossipRelayTest`'s class doc comment for the full mechanism this mirrors.

**Bounded polling throughout, no fixed sleeps.** Both the warm-up phase and the grant-propagation
phase poll against a single deadline computed once at function entry and shared across both phases,
rather than each phase independently claiming its own full timeout budget.

**Test coverage.** `MultiNodeTrustPropagationDemoTest` calls `runMultiNodeTrustPropagationDemo()`
directly (bypassing `main()`'s console output) and asserts real correctness: exact grant convergence
across all three nodes' local views (`grantsOnA`/`grantsOnB`/`grantsOnC` each equal to the single
issued grant), the exact resolved `scoreMicros`, and the `hops shouldBe 1` distinction above.

**Review note on resource cleanup.** The nine `veritas*/pubsub*/storage*.stop()` calls run
sequentially just before `return result`, not duplicated in the `finally` block alongside the three
`node.stop()` calls. This was flagged and traced through review: all three types' `stop()` are
no-ops or purely in-memory today, and `node.stop()` (already in `finally`) unconditionally tears down
the actual bound `Host`/socket regardless of whether the nine calls are reached — so no real leak
exists on an early-exception path, and a nullable-`var` restructuring to defensively cover a already
provably-harmless path was deliberately not made.

This wave's review and security audit both approved with only this one minor/cosmetic finding — no
fix round was needed beyond the explanatory comment described above.

With V0.1.8 landed, the V0.1 milestone (identity, node bootstrap, Nabu storage, Veritas trust model,
trust path-selection, gossip propagation + persistence, and now a demonstrated end-to-end scenario)
is complete.

== Virtus: LTR core (V0.2.1)

`lapis-net-virtus` is the second of the four scoring dimensions (Layering above), and the first
milestone of V0.2. Its data model, `LtrRecord` — a payer-signed claim binding an on-chain OP_RETURN
Bitcoin payment to a `(cid, viewId)` pair — is deliberately a **sibling** to `lapis-net-trust`'s
`VeritasGrant`, not built on top of it: the module has no dependency on `lapis-net-trust`, and the
two are structurally different in the one way that matters most. A Veritas grant forms a
per-`(truster, target)` version chain that gets resolved to one latest edge; an LTR record for the
same `(cid, viewId)` **accumulates independently and is never resolved to one** — every boost is
kept, per the fach-spec's accumulation rule (multiple LTR payments for the same content/view are
independent records, not a merged running total).

**GossipSub propagation, Nabu persistence, and the two-cap index are carried over directly, not
re-derived.** `LtrGossip`/`LtrRecordIndex` reuse the exact three-step validator structure and
two-cap eviction/persistence pattern that took `lapis-net-trust`'s V0.1.7 four security-review
rounds to land on: `canAccept()` (an evicting, LRU-style tracking cap) gates dedup, a separate,
non-evicting `tryReservePersistence()` independently bounds disk writes, and reaching the
persistence cap does **not** decline the underlying gossip message — a record still gets indexed
into the live trust/economy view even when it can't be persisted to disk. Both this wave's
independent code review and its independent security audit specifically verified the carried-over
pattern reproduces the *hardened* (post-fix) Veritas behavior, not an earlier, since-corrected
shape — this is the primary reason `lapis-net-virtus` shipped with no critical/major security
findings in its first review round.

**The `nonce` field is not in the fach-spec's literal schema — an engineering addition, documented
as such.** ECDSA signing here is deterministic (RFC 6979): the same digest signed with the same key
always produces the same signature. Because accumulation means two genuinely independent boosts —
same payer, same `cid`/`viewId`/amount, landing in the same wall-clock second — are both supposed to
be kept, they would otherwise encode to byte-identical records and the second would be silently
discarded by content-id dedup as if it were merely a gossip replay of the first, permanently losing
one real boost's economic weight. An 8-byte `SecureRandom` nonce, generated fresh per `create()`
call and included inside the signed body (so it cannot be stripped or replaced without invalidating
the signature), closes this gap.

**Decay math is `Double`, deliberately unlike Veritas's fixed-point `trustMicros` convention.**
`LtrWeightCalculator.decayedWeightMsat()` implements the fach-spec's formula exactly:
`initial_value × 0.9^(hours_since_timestamp / 24)`. This value is never signed, never part of any
canonical encoded bytes, and never compared for cross-node consensus — it is a local, read-time-only
ranking heuristic each view computes independently for its own sort order, so `Double`/`Math.pow`'s
imprecision is the right tradeoff for a fractional exponent with no clean fixed-point form. A
future-timestamped record (clock skew, malice, or a bug) has its elapsed-hours clamped to zero
rather than going negative — without this clamp, a negative exponent would push the decay factor
*above* 1.0, letting a record's weight exceed its own face value and breaking the "decay is
monotonically non-increasing" invariant the fach-spec's own worked table assumes.
`LtrWeightCalculator.totalInvestedMsat()`, by contrast, stays exact (`Long`, `Math.addExact`-guarded,
throws `ArithmeticException` on overflow rather than silently wrapping) because it *is* a real
accounting figure, not a ranking heuristic — unlike the decayed weight, it is meant to answer "how
many sats have actually flowed to this view," where floating-point imprecision would be wrong.

**Known, deliberately accepted gap: no live on-chain verification.** `OnChainProof` validates only
that a claimed Bitcoin txid is a structurally plausible 32-byte hash and that its output index is a
non-negative value within a defensive sanity cap — it does not verify that a transaction with that
txid exists on-chain, that its output really pays the claimed view's address, or that its
`OP_RETURN` output actually encodes the claimed `cid`/`viewId` pair. This has a sharper consequence
here than the analogous gap does for Veritas: because the fach-spec's whole anti-Sybil claim for LTR
rests on "every boost costs real sats," and this wave cannot check that a payment is real, an
`LtrRecord` today is exactly as cheap to fabricate as a `VeritasGrant` — a malicious node can claim
an arbitrary msat amount was paid, at zero cost, and any view that honors `LtrGossip.currentWeight()`
or `LtrWeightCalculator.totalInvestedMsat()` today is trivially manipulable by anyone willing to run
this code. Closing this gap is real future work (a chain-verification sub-wave, or reusing whatever
SPV/full-node integration the Lightning wave, V0.6, ends up needing anyway) — until then, a
browser/view that wants economically-trustworthy sorting must run its own verification layer on top
of `LtrGossip.currentRecords()`, or knowingly accept this wave's weaker guarantee (cryptographic
attribution to a claimed payer, no proof the payer actually paid).

**No Bitcoin transaction library dependency was added this wave.** `fr.acinq.bitcoin:bitcoin-kmp-jvm`
exists on Maven Central and its POM pins the exact `secp256k1-kmp-jvm` version already used by this
project, so it is a safe candidate for whichever future wave actually constructs, signs, or verifies
real Bitcoin transaction bytes — nothing in V0.2.1 parses or constructs transaction bytes, so adding
the dependency now would be unused weight.

== Minimal-Browser MVP (V0.2.2)

`lapis-net-browser` is the first real Lapis Net client — the V0.2 milestone's second and final wave.
It embeds a real node (identity, GossipSub, Nabu storage, Veritas, Virtus) in the same JVM process as
a small local HTTP server, and serves a hand-written, framework-free HTML/CSS/vanilla-JS page that
shows a timeline of content annotated with Veritas credibility and Virtus/LTR-weighted sort order.

**Tech-stack decision: an embedded local Ktor server, not Compose Multiplatform or KVision/Kilua.**
Lapis Net's sibling federated projects (Lapis Cloud, Pepela Portal) use a Ktor-server-plus-KVision-
web-client split for a real architectural reason those projects have and this one doesn't: a genuine
client/server separation. A `LapisNode` already *is* the full network participant — there is no
separate backend to RPC into. Compose Multiplatform Desktop would be the more architecturally
"correct" long-term fit (direct in-process function calls, no HTTP layer at all), but it is a heavy
new toolchain this project doesn't need yet to answer this wave's actual question ("can a small
pilot group see Veritas/Virtus data and use it"). A tiny embedded Ktor server plus a `fetch()`-only
static page — no JS build toolchain, no npm, no bundler — is the smallest new dependency surface
that gets a pilot test in front of users soonest, without foreclosing a future Compose Desktop (or
web) rewrite once real pilot feedback exists.

**`PostAnnouncement`: small, deliberately temporary content-discovery scaffolding, not a protocol
commitment.** Before any timeline can exist, the browser needs to know which content is even a
candidate — and no signed author-to-content binding exists anywhere else in this codebase.
`NabuStorage` is anonymous, content-addressed byte storage; `VeritasGrant` is a trust edge, not
content; `LtrRecord` only ever *references* an already-existing CID, never creates one. The
mechanism meant to solve "how does content enter the candidate pool" is Karma (`Karma → Veritas →
Virtus`, per `docs/requirements.adoc`) — not built yet (V0.3). `PostAnnouncement`, scoped to
`lapis-net-browser` (application layer, not protocol core or scoring substrate), plays that role
just well enough for one browser vendor's MVP: a small, signed struct mirroring `LtrRecord`'s
hardened codec/gossip/index shape exactly, explicitly documented as scaffolding likely to be deleted
wholesale once V0.3's real Karma lands, not a commitment other vendors must adopt.

**One deliberate structural deviation from `LtrRecord`'s precedent: the post body bytes travel with
the announcement, not just a `Cid` reference.** `NabuStorage`'s own documented V0.1.4 known
limitation is that cross-node DHT provider discovery (`findProviders()`) does not work — a
`PostAnnouncement` that only carried a `Cid` would hit that same broken discovery path the moment a
second node tries to render someone else's post. Embedding `bodyBytes` directly and having every
node — sender and every receiver — independently derive its own `Cid` via `storage.put(bodyBytes)`
sidesteps the gap entirely: content-addressing guarantees identical bytes hash to the identical
`Cid` on every node, with no cross-node discovery step needed at all.

**The `Credibility.NO_PATH` vs. `RESOLVED(0)` distinction is the single most safety-critical piece
of logic in this module.** A brand-new pilot user with no trust relationships must see their whole
timeline as "unrated," never as "everyone is maximally distrusted" — collapsing this distinction
(treating `TrustPathFinder.findPath() == null` the same as a resolved path with score 0) would
directly contradict Veritas's own documented bootstrap design (a newcomer sees the unfiltered
stream, not something that *looks* like universal distrust). `TimelineBuilder`'s credibility filter
only ever applies to a `RESOLVED` score below threshold; a `NO_PATH` candidate is never filtered.

**Review findings this wave — one genuinely critical.** Independent security review found
`PostAnnouncementGossip` wrote every gossiped post's body bytes to storage completely uncapped — no
persistence cap gated this write at all, unlike `LtrRecordIndex`'s hardened two-cap pattern this
module otherwise mirrors faithfully. Since a cheap, self-signed, freshly-nonced `PostAnnouncement`
always passes content-id dedup, and every relaying node independently repeated the same unconditional
write, this reproduced V0.1.7's fixed "network-amplified disk growth" bug class in a worse form (no
cap at all, versus "a cap exists but might be reached"). Fixed with an independent, `Cid`-keyed,
non-evicting `MAX_PERSISTED_BODIES` cap (`PostAnnouncementIndex.tryReserveBodyPersistence()`) gating
the body-bytes write at both the `announce()` and `onGossipMessage()` call sites — reaching the cap
does not decline the announcement itself, only its local body cache, matching this codebase's
established "declining persistence never declines the message" pattern. A parallel major finding —
uncaught storage-fault exceptions leaked full local filesystem paths in HTTP 500 responses,
live-reproduced against both `GET /api/timeline` and `POST /api/posts` — was fixed with a generic
Ktor `StatusPages` handler (defense-in-depth against any future uncaught exception in any route)
plus a safe fallback on the timeline's per-entry storage lookup. Both fixes were independently
re-verified in a second review round, including live re-reproduction of the path-leak scenario
against the fix.

**Known, deliberately accepted gaps:**

* No LTR/Virtus record *authoring* in this MVP — read-only display only. Real LTR spending needs
  actual Bitcoin/Lightning payment (V0.6), and V0.2.1's `OnChainProof` has zero live chain
  verification, so an in-app "spend sats" button today would either lie about a real payment or
  require users to hand-construct a transaction outside the app — both bad pilot UX. Practical
  consequence: Virtus sorting is visually inert (everything at zero weight, sorted by recency)
  during the pilot unless an operator manually seeds demo `LtrRecord`s outside the HTTP surface.
* No "confirm" mechanism (multi-confirmer credibility) — no protocol data model for it exists yet.
  `Credibility`'s `confirmers` parameter is forward-compatible scaffolding, always empty this wave.
* No bootstrap infrastructure exists yet (V0.4) — `POST /api/peers/connect` (manual multiaddr
  exchange, mirroring Veritas's own public-key-exchange convention) is necessary MVP scope, not an
  extra: without it, pilot nodes on different networks could never find each other at all.
* No rate limiting on the signing routes (`POST /api/posts`/`POST /api/trust`) — an accepted
  tradeoff for a loopback-only, single-user pilot server holding the operator's own key. The
  server's `127.0.0.1`-only binding is a hard, tested requirement, not a suggestion; the JSON-only
  `ContentNegotiation` setup incidentally blocks browser-based cross-origin requests to these routes
  today, though this is an incidental side effect of content-type enforcement, not a configured
  CORS policy — worth revisiting explicitly if this ever changes.
* No automated UI/JS test coverage — the actual complexity (scoring, sorting, gossip validation,
  signing) is fully covered by the pure-core and Ktor-route test suites; the HTML/JS itself is a
  thin `fetch()` layer over already-tested JSON endpoints, verified instead by a one-time manual
  two-node checklist (documented in the wave's own commit history) rather than a browser-automation
  dependency.

With V0.2.2 landed, the V0.2 milestone (Virtus/LTR economy plus a real, usable first client) is
complete.

**UI/UX design-team review (retroactive).** The project's house rule requiring every UI/UX task to
go through a fixed design panel (Kare, Tesler, Atkinson, Kay, Norman, Raskin, Rams, Ive, Forstall,
Duarte, Zhuo, with Steve Jobs's final review) was added after the static UI above already shipped —
applied retroactively as a follow-up commit. Seven changes landed: shape+glyph (not hue alone) on
the credibility badge so `NO_PATH` can never be mistaken for a low `RESOLVED` score even under
colorblindness or a fast skim; an always-open on-page explainer for what the badge and LTR boost
mean; an explicit disclosure before a trust rating is submitted that it is a public, signed,
accountable statement; a brief success flash on the timeline item that just changed; a
primary/secondary button hierarchy; numbers paired with plain words ("Trusted · 62%", "Boosted 2.50
sats by 3 supporters"); and a labeled identity ("You:") instead of a bare hex string as the first
thing a first-time reader sees. No framework, build step, or dependency was added — this stayed
within the already-decided hand-written HTML/CSS/vanilla-JS tech stack, and the `textContent`-only
XSS discipline is unchanged. Two real improvements (showing the actual trust path, visualizing LTR
decay over time) were explicitly deferred, not silently dropped — both need new fields in the
`/api/timeline` response that the server doesn't currently send.

== Karma: Bitcoin-anchored one-click likes (V0.3)

`lapis-net-karma` is the third of the four scoring dimensions (Layering above). Its data model,
`KarmaVote`, is a cheap, one-click "like" — but one carrying built-in Sybil resistance from two
independent mechanisms combined: a Bitcoin-chain-derived time anchor and Veritas-weighting.
`Karma = t/(n+1)`, where `t` is the number of Bitcoin blocks elapsed since the voter's own first
outgoing transaction (no chain history ⇒ `t < 0` structurally impossible to represent ⇒ Karma is
always `0.0` for an unanchored identity — enforced by `NoAnchorClaim`/`ChainAnchorClaim`'s own
`init` invariants, not merely by convention), and `n` is the voter's own prior eligible-vote count —
critically, always derived locally by each observer from its own indexed vote history, **never**
self-declared inside the wire message (`KarmaVoteCodec`'s layout has no `n` field at all).

**No live chain-verification in the GossipSub validation path, ever — the wave's central design
decision.** `KarmaGossip.onGossipMessage` validates only structure and signature; it never calls
out to Electrum or any network service. This mirrors `OnChainProof`'s already-accepted V0.2.1
precedent exactly, for the same three reasons: avoiding a liveness dependency on a third-party
server inside the hot gossip-validation path, avoiding cross-node non-determinism (different nodes
could observe different, possibly reorg'd chain state), and avoiding DoS amplification (an attacker
flooding gossip messages that each trigger an outbound network call would be a powerful
amplification primitive). Chain data is only ever consulted client-side, once, at local vote-cast
time, via `KarmaAnchorCache` — never as part of judging someone else's vote.

**Chain-data source: a trusting light client, explicitly not trustless SPV.** No genuine SPV client
library exists on Maven Central for Kotlin/JVM (confirmed during this wave's planning); building
real header-chain-verified SPV from scratch was out of scope. `ElectrumTimeAnchorSource` talks to a
real Electrum server over the `electrum-client`/`bitcoin-kmp` libraries and trusts every response at
face value — history entries, raw transaction bytes, chain tip height — with no merkle-proof
verification against a block header chain and no independent cross-check against a second server. A
malicious or compromised Electrum server can lie about any of these facts undetected. This is an
accepted, explicitly-documented scope limitation, mirroring `OnChainProof`'s own "structural/
trusting, not verified" precedent, and only matters for a node's own outgoing vote — `KarmaGossip`
never calls this class at all. The underlying Electrum protocol transport is also plaintext TCP
only, no TLS, documented as a further caveat of the same "trusting, not verified" shape.

**Bitcoin address derivation: P2WPKH only, a deliberate protocol-defined scope narrowing.** An
identity's Bitcoin presence is looked up via `HASH160(compressed secp256k1 pubkey)` —
`BitcoinAddressDerivation` — the sole derivation this wave recognizes. Legacy P2PKH, other HD
derivation paths, multisig, and Taproot addresses are not found, even if the identity has genuine
Bitcoin history under one of those. This narrows Karma's real-world reach in exchange for a single,
unambiguous, protocol-wide derivation rule every node computes identically.

**Sybil-resistance fix landed during this wave's review loop: at most one tracked vote per
`(voter, targetCid)`.** The original implementation capped `KarmaVoteIndex` only globally
(`MAX_TRACKED_VOTES`), with no per-voter limit. Because `KarmaVote.nonce` makes every vote
content-id-distinct, a single identity could sign and gossip unlimited free votes (no PoW, no
stake, unlike Virtus's `LtrRecord`, which at least claims a real BTC payment) — each one evicting
some unrelated voter's/target's vote from the shared index, and inflating a single target's Karma
via unbounded harmonic-sum repeat-voting (`t/(n+1)` only shrinks the *marginal* value of a repeat
vote; the *total* contribution from `M` repeat votes toward one target grows as `t · H_M`, unbounded
as `M` grows). Fixed by making a new vote from a voter who already has a tracked vote for the same
target **replace** the existing entry (most-recent-wins), keeping all three secondary indices
(`votesByContentId`/`votesByTarget`/`votesByVoterMap`) in lockstep via the same helper used by LRU
eviction. This fully closes per-target Karma inflation from one identity; it deliberately does
**not** limit how many *distinct* targets one identity can vote on (a narrower, lower-severity
residual gap, tracked as future work — see below) — Sybil resistance across many distinct
*identities* is intentionally left to the Bitcoin-anchor cost, not to this index.

**`TimelineBuilder`/`TimelineEntry` scope discipline: Karma is purely additive/display, never
affects sort or filtering.** This is a deliberate narrowing versus the fach-spec's fuller vision of
Karma "bridging trust networks" — `TimelineBuilder` does not yet have a genuine default-private-
timeline mode to bridge past, so wiring Karma into sort order or the credibility filter this wave
would either be a no-op or would silently misrepresent what Karma actually measures for a viewer.
`karmaScore`/`karmaVoteCount` are attached to every timeline entry and rendered, but
`TimelineBuilder.build`'s sort key remains exactly `(ltrWeightMsat desc, publishedAtEpochSeconds
desc)`, and `filteredOut` is still computed purely from Veritas credibility — proven by dedicated
tests showing a huge self-cast karma score changes neither ordering nor filtering.

**Review findings this wave.** Two independent review+security rounds ran before this wave landed
cleanly. Round 1 found and fixed: the per-voter vote-flooding gap above (the wave's one MAJOR
finding, security-relevant); `KarmaAnchorCache` never cached `TimeAnchorLookupResult.NotFound`,
contradicting its own doc comment and re-triggering a full Electrum scan (up to
`maxHistoryEntriesScanned` round-trips) on every single vote cast by an as-yet-unanchored identity
— fixed with a bounded `NOT_FOUND_TTL_SECONDS` cache, unlike a genuine `Found` genesis transaction,
which is a fixed historical fact and is cached without expiry; a missing `KarmaAnchorCacheTest`
covering the cache's security-critical contracts (added, 9 tests); `cachedGenesis` was an
unsynchronized `HashMap` under concurrent Ktor request threads, keyed by a lossy truncated
fingerprint string — fixed by switching to a `ConcurrentHashMap` keyed directly by
`Secp256k1PublicKey`; raw Electrum server error text could leak into the `POST /api/karma` HTTP 502
body — fixed by logging full detail server-side and surfacing only a generic message; a missing CID
length bound on `parseCidOrNull`; and the Like button's "this is a public, signed statement under
your own identity" framing was buried in a collapsed FAQ rather than always visible near the button
itself, unlike the equivalent trust-rating-form notice. Round 2 independently verified all of the
above and surfaced one residual gap in the same error-sanitization fix: the chain-tip-height RPC
path (`ElectrumTimeAnchorSource.currentChainTipHeight()`) used a separate code path that the first
sanitization pass missed, so a malicious Electrum server could still leak raw response text via that
sibling call — closed with the identical treatment (log full detail, throw a generic message) plus a
regression test proving the leak no longer occurs.

**Known, deliberately accepted gaps:**

* **Distinct-target vote flooding is not capped.** The per-target replace-on-conflict fix above
  closes per-target Karma inflation from one identity, but a single costless identity can still
  gossip votes toward many thousands of *different* target CIDs, still consuming
  `KarmaVoteIndex`'s shared `MAX_TRACKED_VOTES` LRU slots and evicting unrelated voters'/targets'
  entries. Tracked as a follow-up — e.g. a per-voter distinct-target quota or a lightweight
  admission cost.
* **No live chain verification anywhere** — see above; this is the wave's central, permanent design
  boundary for the gossip path, not a gap expected to close later the way V0.2.1's `OnChainProof`
  gap might.
* **`RealElectrumRpc`'s connection has no `close()`/lifecycle wiring into `BrowserServer.stop()`** —
  a real, if bounded, resource gap: one live socket + daemon thread persists for the process
  lifetime once a node's first vote establishes it. Not attacker-amplifiable, since nothing external
  can trigger additional `RealElectrumRpc` instances to be constructed.
* **Empty `ElectrumServers.PLACEHOLDER` ships by default** — every real `POST /api/karma` call fails
  with a clean 502 out of the box until an operator configures a real, reachable Electrum server
  list; this sandbox's outbound network policy could not verify real Electrum reachability during
  this wave (raw TCP to well-known public Electrum servers all timed out through the proxy), so an
  unverified hostname list was deliberately not shipped in its place.
* **`n`'s observer-dependent nature** — different observers can legitimately compute a different
  `karmaValue()` for the same vote depending on their own partial gossip view (which prior votes
  from that voter they happen to have indexed). This is by design: `n` is a local heuristic, not a
  consensus or signed value.

With V0.3 landed, all three of Veritas, Virtus, and Karma — three of the fach-spec's four scoring
dimensions — are implemented; Madli remains the only one not yet started.
