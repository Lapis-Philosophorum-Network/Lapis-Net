= Lapis Net Architecture

== Status

V0.1.7: identity layer (V0.1.2), P2P node bootstrap (V0.1.3), DHT + Bitswap
content storage via Nabu (V0.1.4), the Veritas trust-edge grant data model
(V0.1.5), the Veritas transitive trust path-selection algorithm (V0.1.6), and

are implemented. Remaining modules are still skeletons. This document grows
as modules gain real implementations.

== Layering

Lapis Net separates a mandatory protocol core from an optional scoring
substrate and a fully pluggable application layer:

* *Protocol core* (mandatory for every participant): identity (secp256k1),
  content addressing, storage (Nabu/Bitswap), peer discovery (DHT),
  propagation (GossipSub), Lightning payments.
* *Scoring substrate* (optional): Veritas, Karma, Madli as network-wide
  reputation records in the DHT; Virtus/LTR as view-specific records bound to
  a browser vendor.
* *Application layer* (freely swappable per vendor): browser/view
  implementations, sorting, scoring interpretation, UI, ad policy.

The network functions even with zero scoring systems in use — Veritas,
Virtus, Karma, and Madli are tools for views, not a protocol requirement.

== Module map

* `lapis-net-core` — shared abstractions, including the `crypto` package's domain-separated
  digest and public-fingerprint helpers used by all signing code in the project
* `lapis-net-identity` — identity (implemented in V0.1.2, see below)
* `lapis-net-storage` — storage/DHT (Nabu, implemented in V0.1.4, see below)
* `lapis-net-trust` — Veritas web-of-trust: signed trust-edge grant model (V0.1.5), the transitive
  trust path-selection algorithm (V0.1.6), and GossipSub propagation + Nabu persistence (V0.1.7,
  see below) are all implemented
* `lapis-net-networking` — libp2p node bootstrap (V0.1.3) and GossipSub pub/sub (V0.1.7, see below)
* `lapis-net-test` — shared test fixtures and helpers (not yet consumed by other modules' tests)
* `lapis-net-cli` — entry point / demo harness

== Identity (V0.1.2)

The canonical Lapis Net identity is a **secp256k1 keypair** (`secp256k1-kmp`) — the same key type
that would back a Bitcoin wallet or Lightning channel in a later wave. There is no recovery
mechanism: losing the private key means losing the identity permanently, by design, analogous to
Bitcoin.

libp2p natively wants an **Ed25519** keypair for its peer ID. Since secp256k1 is the
real identity, each identity carries a second, Ed25519 keypair (Bouncy Castle,
`bcprov-jdk18on` — the same library jvm-libp2p itself depends on) used only for the libp2p peer
ID, plus a **dual-key binding**: the secp256k1 key signs a domain-separated digest of the Ed25519
public key, cryptographically proving "this libp2p peer ID belongs to this Lapis Net identity"
without making Ed25519 the canonical identity.

Domain separation (`net.lapisphilosophorum.lapisnet.core.crypto.domainSeparatedDigest`) is a
project-wide convention, not specific to identity binding: every distinct purpose that signs a
32-byte digest with a secp256k1 key (identity binding today; Veritas trust-edge grants from
V0.1.5) must use its own versioned ASCII tag, so a signature produced for one purpose can never be
confused with a signature for another purpose using the same key.

Identities persist to a fixed-layout binary keystore file (`KeystoreFileFormat`, one file per
label under `$LAPISNET_HOME/identity` or `~/.lapisnet/identity`), protected only by POSIX file
permissions (`0600` files, `0700` directory) — the same trust model `ssh-keygen` uses by default.
This is a deliberate, temporary tradeoff for an early prototype: passphrase-derived encryption at
rest is a documented follow-up that should land before this key custodies real funds (Lightning).
A user may hold several identities (pseudonyms) side by side, addressed by label — device-sync /
self-trust linking between a user's own pseudonyms is a later wave (see the project's Veritas
notes), not implemented here.

== Networking (V0.1.3)

The node's libp2p identity is derived directly from `DualKeyIdentity`'s Ed25519 keypair
(`deriveLibp2pPeerId()`): the raw 32-byte seed V0.1.2 already stores is exactly the byte layout
jvm-libp2p's key unmarshalling expects, so no adapter or re-encoding is needed. The `PeerId` is
derived from the *private* key (via its public key), never read from a separately-stored public
key, so a corrupted keystore can never desync the `PeerId` from what the node can actually sign
with.

`LapisNode` wraps a jvm-libp2p `Host` built from the production-tested combination the library's
own status table recommends: TCP transport, Noise secure channel, mplex stream muxer. mDNS
discovers LAN peers but never auto-dials them — jvm-libp2p's own `Discoverer` API is
discovery-only by design, and keeping discovery and connectivity independently observable makes
both easier to test. Bootstrap peers are the opposite case: dialing them on `start()` is the
whole point, done non-blockingly and non-fatally so an unreachable peer can never hang or fail
startup. No real bootstrap infrastructure exists yet — `BootstrapPeers.PLACEHOLDER` uses RFC 5737
documentation-range addresses, explicitly marked as a pre-deployment placeholder.

jvm-libp2p is not published to Maven Central; it ships across three mirrors (Cloudsmith, JitPack,
Consensys artifacts) that must all be declared as repositories, since its own transitive
dependencies (`java-multibase`, `noise-java`) are split across them.

GossipSub pub/sub and NAT traversal are explicitly out of scope for this wave; GossipSub lands in
V0.1.7 (see below), NAT traversal remains an open research item.

== Storage (V0.1.4)

`NabuStorage` (in `lapis-net-storage`) layers Nabu's DHT (Kademlia) and Bitswap protocols onto
the same `Host` a `LapisNode` already built and started in V0.1.3 (`LapisNode.host` is exposed,
read-only, specifically for this). Its API is domain-agnostic by design — `ByteArray`/`Cid` in
and out — so it can serve as the storage substrate for signed Veritas records (V0.1.5+) without
knowing anything about Veritas itself.

**Classpath collision with Nabu's own forked jvm-libp2p.** Nabu's `pom.xml` pins a transitive
dependency on `com.github.peergos:jvm-libp2p:0.18.0-ipv6-mdns-wildcard`, a Peergos-maintained fork
of the official `io.libp2p:jvm-libp2p` this project already depends on. The fork ships identical
Java package names (`io.libp2p.core.*`) but different bytecode for 361 of 829 shared classes
(verified by decompiling both jars). `lapis-net-storage/build.gradle.kts` excludes the fork and
forces Nabu to resolve against the official artifact instead — verified safe because Nabu's
`Bitswap`/`Kademlia` classes (and their engines) have zero compiled references to the
`io.libp2p.core.dsl.*` builder classes that actually differ between the two jars (only Nabu's own
unused `org.peergos.HostBuilder` touches those), confirmed both statically and by a real two-node
Bitswap put/get spike against the official jar. `NabuStorage` therefore always attaches to a
`Host` built via the official DSL (`LapisNode.create()`), never via Nabu's own `HostBuilder` or
`EmbeddedIpfs`.

**Access control: none yet, by design.** `NabuStorage.attach()` wires an always-allow
`BlockRequestAuthoriser` — any connected peer can fetch any block this node has stored, provided
or not, the standard Bitswap/IPFS trust model (CIDs are unguessable 256-bit content hashes; a peer
cannot induce this node to fetch-and-forward third-party content, since `get()`/`findProviders()`
are only called by this project's own local code, never in response to a remote request).
Per-caller/per-record access control is a Veritas-layer concern and lands in V0.1.5+, not here.

**DHT record-store bound.** Nabu's own `RamRecordStore` (used for IPNS-style key/value records)
is an uncapped `HashMap`, and its `KademliaEngine` hands any connected peer's signed `PUT_VALUE`
record straight into it with no other rate limit - reachable simply by wiring up Kademlia at all,
regardless of whether this project's own API ever publishes an IPNS record itself.
`lapis-net-storage`'s own `BoundedRecordStore` (LRU-capped, mirroring Nabu's own
`RamProviderStore`) is used instead to prevent unbounded remote-triggered heap growth.

**Known limitation: cross-node DHT provider discovery does not currently work.**
`NabuStorage.provide()`/`findProviders()` wrap Nabu's real `Kademlia.provideBlock`/
`findProviders` API correctly (verified via `javap` against the real jar), and the underlying
DHT routing-table bootstrap (`connectToDhtPeer`, wrapping `Kademlia.bootstrapRoutingTable`) is
verified working — a 3-node loopback chain reliably populates each node's routing table with its
explicitly-connected peer. However, the actual provider announcement/discovery round trip does
not: a standalone diagnostic spike (four isolated runs, outside the test suite, against real
loopback nodes) found that a `GET_PROVIDERS` RPC returns an empty provider list even via Nabu's
own trivial "I already have this block locally" auto-provide path in
`KademliaEngine.receiveRequest`'s `GET_PROVIDERS` case — i.e. this reproduces with zero
involvement of `provide()`/`ADD_PROVIDER` at all, so the fault is in the RPC/dial mechanism
`Kademlia` uses for both message types (`Kademlia.dialPeer`), not specific to provider
announcement. The failure is reproducible, not flaky (unlike V0.1.3's mDNS multicast issue,
which has a genuine external cause) — root cause not identified within this wave's practical
debugging budget; suspected candidates include an interaction between `/ipfs/` vs. `/p2p/`
peer-id multiaddr component naming (routing-table entries carry `/ipfs/`-form addresses) and
`dial()`'s peer-id validation, or a Nabu v0.8.0-specific issue. `lapis-net-storage`'s test suite
pins this with an explicit "KNOWN ISSUE" test
(`MultiNodeDhtProviderDiscoveryTest`) asserting the current (broken) empty result, so a future
fix shows up as a test failure prompting an update, rather than the regression going unnoticed.
What remains fully verified and usable for V0.1.5+: local put/get, and Bitswap fetch given an
explicit peer (the `TwoNodeBitswapDirectFetchTest` path) — i.e. `get(cid, peers = ...)` with a
peer set obtained by some other means (a future wave's own peer-list, GossipSub-announced peers,
etc.) is safe to build on; relying on `findProviders()` to return non-empty is not, until this is
root-caused.

== Trust: Veritas trust-edge grant (V0.1.5)

`VeritasGrant` (in `lapis-net-trust`) is the signed, versioned data model for a single Veritas
web-of-trust edge: a truster's secp256k1-signed statement of how much it trusts a target identity.
It is a pure data-model wave — network propagation (GossipSub) and persistence (Nabu) are
implemented in V0.1.7, below; the shortest-path aggregation algorithm that turns a graph of grants
into a trust score between two arbitrary identities is implemented in V0.1.6, below.

**Fixed-point trust value.** Trust is represented as `trustMicros: Int` in `0..1_000_000`, not as a
`Double`. A signed, cross-platform-verified structure cannot depend on floating-point arithmetic
being bit-identical across JVMs/architectures; a fixed-point integer sidesteps that class of bug
entirely. `trustFraction` (`trustMicros / 1_000_000.0`) exists only as a display-only convenience,
never as part of the signed or compared representation.

**Single target per grant.** Each `VeritasGrant` names exactly one `(truster, target)` pair, not a
batch of targets, because the Git-like backward-hash version chain (below) is intrinsically
per-pair: a grant's `previousGrantId` must unambiguously identify the one prior grant it supersedes
for that same truster/target combination.

**Git-like backward-hash version chain.** Each new grant optionally embeds the content hash
(`VeritasGrantCodec.contentId()`, plain SHA-256 over the full signed-and-signature-appended
encoding) of the grant it supersedes, as `previousGrantId`. The first grant in a chain (no
predecessor) is genesis, represented as a `null` `previousGrantId` with an explicit presence flag
bit in the wire encoding — not a sentinel all-zero hash. Revocation is not a distinct operation or
message type; it is simply a new grant with `trustMicros = 0` that links to its predecessor like
any other update. `VeritasGrantChain.validate()` checks an ordered list of grants forms a valid
chain (constant truster/target, first is genesis, every signature verifies, each grant's
`previousGrantId` matches its predecessor's `contentId()`) — purely in-memory, no I/O; fetching the
actual chain from storage is a V0.1.7 concern.

**Domain-separated signing.** The signed body is digested with
`domainSeparatedDigest("LapisNet:veritas-trust-edge:v1", body)` before signing — its own reserved,
versioned tag distinct from identity binding's `"LapisNet:identity-binding:v1"`, per this project's
domain-separation convention (see Identity above): a signature produced for one purpose can never
be confused with a signature for another purpose using the same key.

**Codec.** `VeritasGrantCodec` uses a sequential length-prefixed binary layout (magic, version,
flags, truster/target public keys, `trustMicros`, optional `previousGrantId`, length-prefixed
comment, length-prefixed occasion-reference CIDs, signature) rather than `KeystoreFileFormat`'s
fixed-offset layout, since several fields are variable-length. `decode()` is structural-only and
deliberately does **not** verify the signature — unlike `KeystoreFileFormat.decode()`, which must,
because a keystore has to be trustworthy just to load. A grant typically arrives from an untrusted
peer over the network, so callers must explicitly call `VeritasGrant.verify()` (or the overload
taking an expected truster) before trusting a decoded grant.

**Security fix: secp256k1 public-key curve validation.** Independent security review for this wave
found that `Secp256k1PublicKey` (in `lapis-net-identity`, present since V0.1.2) validated only that
its input was 33 bytes long, never that those bytes actually encode a point on the secp256k1
curve. The native `Secp256k1.verify()` call throws an uncaught exception on a curve-invalid public
key rather than returning `false`, and does so before any signature check — no valid signature
knowledge is needed to trigger it. This had gone unnoticed through V0.1.2–V0.1.4 because nothing
until this wave constructed a `Secp256k1PublicKey` directly from untrusted, decoded bytes;
`VeritasGrantCodec.decode()` is the first code path that does, making this a real remote-DoS vector
once grants start arriving from peers (V0.1.7). Fixed by validating via `Secp256k1.pubkeyParse()`
in the constructor, converting any native failure into a clean `IllegalArgumentException`; the fix
applies project-wide, since the class is shared. `VeritasGrantCodec.decode()` was also hardened to
reject non-zero reserved flag bits, so tampered bits can no longer silently round-trip as an equal
grant.

== Trust: transitive path selection (V0.1.6)

`TrustGraph` and `TrustPathFinder` (in `lapis-net-trust`) compute a transitive trust score and
winning path between a source and a target identity, over a graph of `VeritasGrant`-derived edges.
This wave is pure algorithm: it consumes an already-resolved edge set (exactly one latest grant per
`(truster, target)` pair) and does not itself fetch grants from storage or resolve a grant's
backward-hash version chain to its latest entry — both are implemented in V0.1.7, below.

**Selection rule: shortest path first, then lexicographic tiebreak — not maximum product.** Trust
along a path is the product of its edge weights, but when multiple shortest (fewest-hop) paths
exist, the winner is not the one with the highest product; it is the one whose hop-weight sequence,
compared from the source outward, is lexicographically greatest (highest first hop wins; ties are
broken by the second hop, then the third, and so on). This models "the closer a person is to me,
the better I can judge them" — a strong nearby relationship outweighs a clever construction of
distant ones, and it is also manipulation-resistant: an attacker cannot inflate transitive trust by
stacking high weights far out in the graph, since only the near edges are ever decisive. The
project's canonical worked example (Odin → Frigg/Thor/Nanna → Freya, where the highest-product path
via Nanna does *not* win because its first hop is weaker) is reproduced verbatim as a binding test
fixture. Self-trust (`source == target`) is axiomatic (`1.0`); no path is `null` (default distrust,
`0`); a zero-trust edge is a real hop that can legitimately be part of, and even win, the shortest
path — there is no fallback to a longer, higher-trust alternative, matching the spec's explicit
single-winner, no-aggregation design for V0.1.

**Fixed-point score, `Long` accumulator.** The path score is computed as `trustMicros: Int` in the
same fixed-point domain as `VeritasGrant.trustMicros`, via a `Long` intermediate accumulator
(`acc = acc * weight / MAX_TRUST_MICROS` at each hop) — two `Int` values up to `1_000_000` multiply
to roughly `10^12`, well past `Int.MAX_VALUE`, so the accumulator must widen even though the final
result always fits back into an `Int`.

**BFS with predecessor pointers, not eager path copies.** Finding "shortest path" means fewest
hops, not lowest weighted distance — this is plain level-synchronous BFS, not Dijkstra. The
algorithm's first implementation stored a full copy of the path-so-far (node list + weight list) at
every node reached during the traversal; on a simple linear chain of `N` nodes this cost `O(N^2)`
time and memory for a single query, and since Veritas grants are free to issue and self-signable in
bulk, an attacker needed no cooperation from anyone else to construct such a chain — a real
remote-DoS vector for the moment gossiped/stored grant data starts feeding real queries (V0.1.7).
The fix replaces the eager copy with a predecessor-pointer scheme: each node stores only a pointer
to its winning predecessor plus the single edge weight into it (`O(1)` per node), and the winning
path is reconstructed exactly once, by walking predecessor pointers backward from the target, only
after BFS finds it. The lexicographic tiebreak (needed when multiple predecessors reach the same
node at the same BFS depth) is implemented as a pair of small recursive comparators that walk
already-finalized predecessor chains, comparing the earliest (most significant) hop first with
early exit on the first difference, rather than comparing pre-built arrays.

**Constant-factor hardening.** A follow-up independent security audit found that while the
predecessor-pointer redesign is asymptotically correct (`O(edges * MAX_HOP_DEPTH)`, confirmed by an
independent hand-trace), the constant factor at the original defense-in-depth caps
(`MAX_NODES = 50_000`, `MAX_EDGES = 200_000`) was not negligible: an adversarial graph where every
edge shares the same weight forces every lexicographic comparison to recurse through the full
predecessor chain before resolving (via a node-public-key-bytes tiebreak) instead of short-
circuiting on the first differing hop — costing several hundred milliseconds per query at those
caps, cheaply reachable since the attacker only needs free, self-signed grants. Two changes
resolved this: `Secp256k1PublicKey.hashCode()` (in `lapis-net-identity`) is now cached at
construction rather than recomputing a 33-byte content hash on every call — safe because the class
is immutable, and a meaningful win since every recursive predecessor-chain comparison hits it via
`HashMap` lookups; and the caps were tightened to `TrustGraph.MAX_NODES = 8_000`,
`TrustGraph.MAX_EDGES = 32_000` (`TrustPathFinder.MAX_HOP_DEPTH` stayed at `20`, itself independently
justified by trust decay: even a generous uniform `0.9` weight per hop decays to `0.9^20 ≈ 0.12` by
that depth, so longer paths are already close to meaningless as trust signals). Re-measured against
the worst adversarial construction found across two independent audit rounds — a tie-heavy graph
shifted to the maximum recursion depth the hop cap allows, not just concentrated at the first two
hops — worst-case cost at the new caps is roughly 35-50ms, an order of magnitude better than at the
original caps and comfortably within the caps' own generous-for-personal-scale headroom (Dunbar's
number is roughly 150; 8,000 nodes is far beyond any realistic personal web-of-trust).

This wave went through three rounds of independent review/security audit before landing — the
correctness of the BFS/tiebreak algorithm itself was hand-traced and approved in the first round and
never needed to change again; both follow-up rounds addressed resource-exhaustion cost at scale, not
correctness.

== Trust: propagation and persistence (V0.1.7)

`GossipPubSub` (in `lapis-net-networking`) and `VeritasGossip`/`VeritasGrantIndex`/
`VeritasGrantResolver` (in `lapis-net-trust`) close the loop V0.1.3-V0.1.6 left open: a node can now
announce a locally-created `VeritasGrant` to its peers over a dedicated GossipSub topic, receive and
validate grants announced by others, persist them into `NabuStorage`, and resolve the "latest grant
per `(truster, target)` pair" edge set that `TrustGraph.fromGrants()` expects.

**Layering.** `GossipPubSub` is domain-agnostic (raw topic strings, `ByteArray` in/out) and lives in
`lapis-net-networking`, layered onto an already-`start()`-ed `LapisNode`'s `Host` exactly the way
`NabuStorage` layers Bitswap/Kademlia onto the same `Host` in V0.1.4. `VeritasGossip` is the
Veritas-specific wiring on top: topic naming, structural decode, signature verification, and the
persistence/resolution logic. `lapis-net-trust` gained a new `api(project(":lapis-net-storage"))`
dependency this wave, since `NabuStorage` now appears directly in `VeritasGossip`'s public factory
signature.

**Two separate signature layers that must never be confused.** GossipSub has its own message-layer
authentication (the `from`/`seqno`/`signature` RPC fields), signed with the node's libp2p Ed25519
identity key (`Host.privKey`) — this is entirely separate from a `VeritasGrant`'s secp256k1 truster
signature, and `GossipPubSub` never touches the latter.

**Full grant bytes are published, not a CID pointer.** A `VeritasGrant`'s encoded size
(`VeritasGrantCodec.MAX_BODY_SIZE` plus a 64-byte signature) is tiny relative to any reasonable
gossip message-size ceiling, and — decisively — `NabuStorage`'s documented V0.1.4 known limitation
is that cross-node DHT provider discovery (`findProviders()`) does not work: a node receiving only a
bare CID over gossip would have no reliable way to fetch the actual content. Publishing full bytes
sidesteps that broken discovery path entirely.

**The GossipSub validator gates both structural and cryptographic acceptance in one step.**
`VeritasGossip.onGossipMessage()` decodes (`VeritasGrantCodec.decode()`), then verifies
(`VeritasGrant.verify()`), before any side effect — deliberately not split into "validate then
separately accept", since GossipSub's own re-propagation to the rest of the mesh is gated purely by
the validator's returned `ValidationResult`; deferring signature verification to a later step would
mean every relaying node re-gossips signature-invalid garbage before individually declining to act
on it.

**Latest-grant-per-pair resolution.** `VeritasGrantResolver.resolveLatest()` collapses a (possibly
forked, possibly byzantine) set of candidate grants for one `(truster, target)` pair down to a
single winner: only signature-valid candidates are considered; every valid genesis grant is a root;
an iterative depth-first walk (explicit stack, not recursion, to avoid stack-overflow risk at
`MAX_CHAIN_DEPTH = 4_096`) follows the backward-hash chain forward from each root, tracking a
visited set to guard against (cryptographically infeasible, but defended anyway) cycles; the winner
among all resulting tip candidates is the one at the greatest depth, tied-broken by
lexicographically-greatest content-id bytes. This tiebreak is deliberately receipt-order-independent
— it depends only on the final candidate set, never on the order candidates arrived via gossip — so
two nodes that eventually observe the same grants via differently-ordered deliveries converge on the
identical resolved graph.

**Known, deliberately accepted gaps** (matching this project's established practice of documenting
limitations rather than silently shipping them):

* The in-memory `VeritasGrantIndex` is never rebuilt by scanning Nabu's local blockstore at startup
  — `NabuStorage` has no "enumerate local CIDs" primitive. A restarted node starts with an empty
  index regardless of what it durably persisted in a previous run.
* A node joining after a grant's gossip window has passed has no backfill path — gossip carries full
  bytes but only propagates live, never historically. This is best-effort convergence, not
  always-eventually-consistent.
* Unbounded message *rate* from a single peer is not defended against at the application layer —
  deferred to jvm-libp2p's own GossipSub mesh-control-message rate limiting.

**This wave went through four rounds of independent review/security audit — the most of any wave so
far** (V0.1.6's three-round precedent was the previous high-water mark), because it's the first wave
where `VeritasGrant`s actually arrive from untrusted network peers rather than being constructed
locally for tests:

* *Critical:* `VeritasGossip.onGossipMessage()` called `VeritasGrant.verify()` unguarded. A
  structurally-right-length but cryptographically degenerate 64-byte signature (e.g. all-`0xFF`
  bytes, or an `r` component at/above the secp256k1 curve order) makes the native
  `Secp256k1.verify()` call throw an uncaught exception instead of returning `false` — the same bug
  class as V0.1.5's `Secp256k1PublicKey` curve-validation fix, but that fix only ever hardened
  public-key *construction*, never signature *verification*. Live-repro'd against the real
  `secp256k1-kmp-jvm` jar. Fixed at the shared root cause — `Secp256k1PublicKey.verify()` (in
  `lapis-net-identity`) now wraps the native call in a catch-all, converting any failure into a
  clean `false` — rather than patching just the new call site, so every current and future caller is
  safe by construction.
* *Critical:* `NabuStorage.put()` ran unconditionally for every signature-valid grant, and
  `VeritasGrantIndex.add()`'s return value was discarded — meaning the index's capacity cap bounded
  neither disk writes nor GossipSub re-propagation (`ValidationResult.Valid` is what triggers every
  relaying node's own mesh-wide re-broadcast). Since Veritas grants cost nothing to issue, a cheap
  attacker could force unbounded, network-amplified disk growth across every node in the mesh.
* *Major:* `GossipPubSub.attach()` left GossipSub's own RPC-frame-shape limits
  (`maxPublishedMessages` etc.) at jvm-libp2p's actual default of "no limit" (confirmed by
  decompiling the real jar) — one peer could pack thousands of distinct grants into a single RPC
  frame, forcing unbounded decode+verify work per frame. Fixed by explicitly bounding
  `maxPublishedMessages`/`maxSubscriptions` via `GossipParamsBuilder`.
* *Major:* `VeritasGrantIndex` had no eviction once full — a cheap attacker filling the shared cap
  could permanently prevent a node from ever tracking new legitimate grants, including a legitimate
  revocation of a compromised or malicious truster (per the domain model, revocation is just a new
  grant with `trustMicros = 0`, not a distinct operation). Fixed with LRU-style eviction, mirroring
  `lapis-net-storage`'s `BoundedRecordStore` pattern.
* *The most interesting finding:* fixing the eviction gap (above) silently reopened the disk-growth
  gap (above it) — once the index stopped hard-rejecting at capacity and started evicting instead,
  its `canAccept()` check no longer had any "index is full" case left to report, so gating
  `storage.put()` on it alone stopped bounding persistence at all. This was caught independently by
  two review agents via two different methods (a hand-trace and an executed repro with `maxTracked =
  2` and 10 distinct grants, all 10 of which landed on disk) converging on the identical finding in
  the same round. Resolved by decoupling the two concerns: the in-memory trust-view index
  (`grantsByContentId`/`grantsByPair`, read exclusively by `TrustGraph`/`currentGrants()`) keeps its
  evicting policy — never permanently locking out legitimate updates is the correct policy for a live
  view — while a new, separate, non-evicting, hard-capped structure (`persistedContentIds`, gated via
  `VeritasGrantIndex.tryReservePersistence()`, cap `MAX_PERSISTED_GRANTS = 64_000`) independently
  bounds whether a durable disk copy gets written. A grant that clears the persistence cap still gets
  indexed and still propagates — this node's own disk budget has no bearing on trust-view correctness
  or mesh-wide convergence, since neither ever reads from disk.
