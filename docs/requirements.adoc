= Lapis Net Requirements Coverage

This document tracks Lapis Net's requirements against what the V0.1 wave
plan (see link:roadmap.adoc[roadmap.adoc] and
link:architecture.adoc[architecture.adoc]) actually covers, and what a
proposed V0.2+ wave plan would need to close. It complements the prose
roadmap with a formal, machine-readable trace: which requirements V0.1
already satisfies, which ones a later wave still needs to satisfy, and how
the still-open requirements depend on each other.

The diagram is authored in kUML and embedded via the `[kuml]` Asciidoctor
macro, per this project's documentation convention (see
xref:architecture.adoc#_diagrams[Diagrams] in architecture.adoc) — the
`kuml-asciidoc` rendering pipeline is not yet wired into this repository's
build, so this source does not render inline yet; it is committed as the
authoritative, version-controlled model regardless.

== Reading the diagram

* **`satisfy` edges** (from `LapisNetV01`) mark what V0.1 (through wave V0.1.7)
  actually delivers: identity (`LN-101`), storage (`LN-102`), propagation
  (`LN-103`), and Veritas (`LN-201`). Every other leaf requirement has no
  `satisfy` edge — that is the visual definition of "not yet built."
* **`contains` edges** mirror this document's grouping: protocol core,
  scoring substrate (the four dimensions), application layer, communication
  subsystems, and identity/bootstrap hardening.
* **`derive` edges** capture real dependencies between still-open
  requirements, taken directly from the research notes (e.g. Karma's voter
  weighting depends on Veritas; the reference browser's timeline pipeline
  depends on all three scoring dimensions it renders; keystore encryption
  must land before Lightning custody, not after).
* **`verify`** ties the pilot community test from the research roadmap
  (testing with a small group — PdV, Girchi, the libertarian scene) back to
  the top-level vision requirement.
